Human Style Coding Standards

Every file name, variable name, function name, class name, database table, error message, and comment in TAVP must be readable by a human who has never seen the code before. If a non-programmer can't roughly understand what something does by reading its name, the name is wrong.

Why this matters: The founder of TAVP is not a programmer. The AI writes 100% of the code. But the founder reads, reviews, and debugs that code. If the code looks like alien language, the founder can't direct the AI, can't review changes, and can't understand what's happening. Human-style code = the founder stays in control of his own product.
01 — PHILOSOPHY

The Core Rule

"If you have to explain the name, the name is wrong."

Code is not written for computers. Computers don't care about names. Code is written for humans — the person who reads it tomorrow, the person who debugs it at 2am, the non-programmer founder who needs to understand what changed.

The Two Tests

Test 1: The Glance Test

Can you understand what this does by glancing at it for 3 seconds?

If yes = human style. If no = rewrite it.

Test 2: The Non-Programmer Test

Could a non-programmer (with basic English) guess what this does?

If yes = human style. If no = rewrite it.

What This Applies To

  • File names — every PHP, Volt, JS, CSS, config, migration, seeder, test file
  • Folder names — every directory in the project
  • Variable names — every $variable in every file
  • Function/method names — every function and class method
  • Class names — every PHP class
  • Database table names — every table and column
  • Route names and URLs — every endpoint
  • Config keys — every key in every config file
  • Error messages — every message shown to users or in CLI
  • Comments — every comment in the code
  • CLI output — every line the tavp command prints
02 — FILE NAMES

File Naming Convention

Rule: File names should describe exactly what's inside. A developer opening the file should know what to expect before reading a single line.

2.1 PHP Files (Classes)

PSR-4 convention: class name = file name. So the class name IS the file name. Make class names human (see Section 5).

Programmer Style (DON'T)
UsrAuthSvc.php // what's "Svc"? OtpDispJob.php // what's "Disp"? UsrPrfCtrl.php // abbreviations MwCsrf.php // "Mw" = middleware? who knows TblUsrMigration.php // "Tbl" prefix = table? AuthRepo.php // "Repo" = repository?
Human Style (DO)
AuthenticationService.php // self-explanatory SendOtpToEmailJob.php // describes exactly what it does UserProfileController.php // clear, no abbreviation VerifyCsrfToken.php // reads like English CreateUsersTable.php // obvious purpose UserRepository.php // full word, no shortcuts

2.2 View Files (.volt)

DON'T
pg_home.volt // "pg" prefix? usr_dash.volt // abbreviations cmp_btn.volt // "cmp" = component? mod_login.volt // "mod" = modal? p_list.volt // p = product? post? page?
DO
home.volt // just "home" dashboard.volt // just "dashboard" button.volt // just "button" login-modal.volt // hyphen, descriptive product-list.volt // full words

2.3 Migration Files

DON'T
2025_07_08_043217_create_users_table.php 2025_07_08_043218_add_col_usr_tbl.php 2025_07_09_011542_tbl_posts.php
DO
001_create_users_table.php 002_create_user_sessions_table.php 003_create_posts_table.php 004_add_avatar_to_users_table.php 005_create_orders_table.php

Simple sequential numbers (001, 002, 003) instead of timestamps. Easy to read, easy to sort, easy to understand order. Timestamps add nothing for humans.

2.4 Config Files

config/app.php // app settings config/database.php // database connections config/mail.php // email settings config/auth.php // authentication settings config/cache.php // cache settings config/queue.php // queue settings config/storage.php // file storage settings config/search.php // search engine settings config/broadcasting.php // real-time broadcasting settings config/phalcon.php // framework-specific settings

2.5 JavaScript Files

DON'T
app.min.js utils.js u.js mod.js
DO
app.js helpers.js modal.js otp-form.js datatable.js

2.6 Test Files

DON'T
AuthTest.php OtpTest.php UsrTest.php
DO

03 — VARIABLES

Variable Naming Convention

Rule: Variables use snake_case. Full English words. No abbreviations. The name must describe what the variable holds.

3.1 General Variables

DON'T
$u // user? url? unicorn? $em // email? error message? $tot // total? toast? $cnt // count? content? cent? $res // result? response? resource? $tmp // template? temp? temporary? $cb // callback? checkbox? code block? $err // lazy abbreviation $isAuth // camelCase (wrong for PHP in TAVP) $currentProductPrice // camelCase (wrong for PHP in TAVP)
DO
$user // clear $user_email // specific $total_price // descriptive $post_count // obvious $api_response // no ambiguity $email_template // full words $form_callback // no confusion $error_message // full word $is_authenticated // snake_case $current_product_price // snake_case, full words

3.2 Boolean Variables

Booleans MUST start with is_, has_, can_, or should_.

DON'T
$active // is it active? or the active record? $verified // is it verified? or the verified value? $auth // is authenticated? or the auth object?
DO
$is_active // clearly boolean $is_verified // clearly boolean $is_authenticated // clearly boolean $has_posts // clearly boolean $can_edit // clearly boolean $should_redirect // clearly boolean

3.3 Collection Variables

Collections (arrays, lists) use plural nouns.

DON'T
$user // is it one user or a list? $data // what data? how much? $items // items of what? $result // single or multiple?
DO
$user // single user $users // list of users $recent_posts // list of recent posts $product_list // list of products $search_results // list of results

3.4 Summary Table

SituationWrongRightWhy
User email$em, $e$user_emailFull words
Post count$cnt, $pc$post_countNo abbreviation
Is logged in?$auth, $il$is_authenticatedBoolean prefix + full word
List of orders$data, $o$order_listDescriptive plural
Total price$tot, $tp$total_priceObvious
Error message$err, $e$error_messageNo shortcuts
Current page$pg, $cp$current_pageClear
04 — FUNCTIONS

Function & Method Naming

Rule: Functions use snake_case. Name = verb + noun. Must describe WHAT it does and (when relevant) WHAT it does it to.
DON'T
function proc($d) { // proc? process? procedure? function handle() { // handle what? function get() { // get what? function send() { // send what to whom? function validate($v) { // validate what? function persist($e) { // persist what? "e" = entity? function dispatchOtp($u) { // camelCase + abbreviation function fetchAll() { // fetch all what? function calc($a, $b) { // calc what? a and b what?
DO
function process_payment($payment_data) { function handle_login_request($request) { function get_user_by_email($email) { function send_otp_to_email($email) { function validate_user_input($input_data) { function save_new_user($user_data) { function send_otp_to_phone($phone_number) { function get_all_published_posts() { function calculate_total_price($items) {

4.1 Common Patterns

ActionPatternExample
Get one thingget_[thing]_by_[criteria]get_user_by_email($email)
Get many thingsget_all_[things]get_all_published_posts()
Createcreate_new_[thing]create_new_user($data)
Updateupdate_[thing]_[what_changed]update_user_profile($user, $data)
Deletedelete_[thing]_by_[criteria]delete_post_by_id($post_id)
Checkis_[condition]is_email_already_registered($email)
Sendsend_[what]_to_[whom]send_welcome_email_to_user($user)
Processprocess_[what]process_payment($payment_data)
Validatevalidate_[what]validate_login_input($request)
Formatformat_[what]_as_[format]format_price_as_currency($price)

4.2 Class Methods (when context is clear)

Inside a class, you can shorten when the subject is obvious from the class name:

// Inside class AuthenticationService: public function send_otp($email) // OK — "to email" is the parameter public function verify_otp($email, $code) // OK — clear from parameters public function login($email, $code) // OK — simple and clear public function logout() // OK — obvious public function current_user() // OK — returns current user // NOT OK (too vague even in context): public function handle() // handle what? public function process() // process what? public function run() // run what?
05 — CLASSES

Class Naming Convention

Rule: Classes use PascalCase. Full English words. No abbreviations. The class name describes WHAT IT IS, not what framework pattern it follows (unless that helps clarity).
DON'T
class UsrAuth {} class OtpSvc {} class PhController {} class MwAuth {} class RepoUser {} class DtoProduct {} class VldEmail {}
DO
class AuthenticationService {} class OtpService {} class BaseController {} class VerifyAuthentication {} class UserRepository {} class ProductData {} class EmailValidator {}

06 — DATABASE

Database Naming Convention

6.1 Table Names

DON'T
usr // abbreviation tbl_users // tbl_ prefix (redundant) userTbl // camelCase + abbreviation m_user // m_ prefix = master? phalcon_users // framework prefix (unnecessary)
DO
users // simple, plural, lowercase user_sessions // descriptive, snake_case password_reset_tokens // self-explanatory payment_transactions // full words social_accounts // clear

6.2 Column Names

DON'T
fn // first name? full name? function? em // email? emergency? dob // date of birth (who knows?) ph // phone? placeholder? pH level? act // active? action? account? dt_cre // date created (cryptic) flg_del // flag deleted (old-school)
DO
first_name // full words, snake_case email // just "email" date_of_birth // fully spelled out phone_number // no ambiguity is_active // boolean prefix created_at // standard, clear deleted_at // nullable timestamp (soft delete)

6.3 Naming Rules

RuleExample
Tables: plural, lowercase, snake_caseusers, blog_posts, payment_transactions
Columns: lowercase, snake_case, full wordsfirst_name, created_at, total_amount
Booleans: is_ prefixis_active, is_verified, is_featured
Foreign keys: singular_table_iduser_id, post_id, order_id
Timestamps: _at suffixcreated_at, updated_at, deleted_at, published_at
Counts: _count suffixview_count, download_count, login_count
Amounts: total_ or _amount suffixtotal_price, discount_amount, tax_amount
07 — COMMENTS

Comment Style

Rule: Comments explain WHY, not WHAT. If the code is so unclear that you need to explain WHAT it does, rewrite the code instead.
DON'T (explaining what the code does)
// Loop through users foreach ($users as $user) { ... } // Check if email is set if ($user_email) { ... } // Get the total $total = $price * $quantity; // Return the response return $response;
DO (explaining WHY)
// We send OTP to unverified users only — verified users use password login foreach ($unverified_users as $user) { ... } // Email can be empty when user registered via phone number if ($user_email) { ... } // Tax is included in the total for Indonesian customers (PPN 11%) $total = $price * $quantity * 1.11; // 204 = success with no body (used after DELETE operations) return $response->setStatusCode(204);

7.1 When to Comment

  • Business rules — "Why" a calculation or condition exists
  • Non-obvious decisions — "Why" we chose this approach over another
  • External dependencies — "Why" we depend on this specific service/version
  • Edge cases — "Why" this check exists (what would break without it)
  • TODO markers — "What" needs to be done later (with context)

7.2 When NOT to Comment

  • When the code is self-explanatory (variable/function names are clear)
  • When the comment just restates the code in English
  • When the comment says "TODO" with no context
08 — ERROR MESSAGES

Error & CLI Message Style

Rule: Every error message must tell the user (1) what went wrong, (2) why, and (3) what to do next. A non-programmer must be able to understand and act on every message.

8.1 User-Facing Errors

DON'T
"ValidationException: email.invalid" "OTP_EXPIRED" "Error 422" "Query failed" "ERR_AUTH_001" "Something went wrong"
DO
"Please enter a valid email address (example: name@company.com)" "Your OTP code has expired. Please request a new one." "We couldn't save your information. Please check the highlighted fields." "We couldn't load the product list. Please try refreshing the page." "Login failed. The OTP code is incorrect. You have 3 attempts remaining." "Something went wrong on our end. Our team has been notified. Please try again in a few minutes."

8.2 CLI Messages

DON'T
ERR: Module not found FAIL PHP Fatal error: Uncaught... Process exited with code 1
DO
✗ Module "tavp/cms" is not installed. Run: tavp module:install tavp/cms Or visit: https://docs.tavp.dev/modules/cms ✗ Installation failed. Phalcon 5.16.0 requires PHP >= 8.1 Your server has PHP 7.4 installed. Solution: upgrade PHP or use an older Phalcon version. Guide: https://docs.tavp.dev/install/php-upgrade

8.3 Error Message Template

// Every error message follows this structure: // USER-FACING (shown in browser/app): [What went wrong in plain language] [What the user should do to fix it] // CLI-FACING (shown in terminal): ✗ [What went wrong] [Why it happened — technical but readable] [What to do: command or link] // API-FACING (returned as JSON): { "success": false, "error": { "message": "Please enter a valid email address", "code": "invalid_email", "details": { "email": "The value 'not-an-email' is not a valid email" } } }
09 — ROUTES

Route & URL Naming

DON'T
/api/v1/usr/otp/dispatch /admin/usrmgmt /api/v1/auth/vld /adm/posts/edt/1
DO
/api/v1/auth/send-otp /admin/users /api/v1/auth/verify-otp /admin/posts/1/edit

9.1 URL Rules

  • lowercase — always
  • kebab-case — send-otp not send_otp or sendOtp
  • plural nouns — /users, /posts, /products
  • verbs for actions — /send-otp, /verify-email, /publish
  • no abbreviations — /users not /usr, /products not /prd

9.2 Route Names

// Route names use dot notation, descriptive: Route::get('/login')->name('auth.login') Route::post('/send-otp')->name('auth.send_otp') Route::get('/dashboard')->name('admin.dashboard') Route::resource('/posts')->name('admin.posts') Route::get('/api/v1/products')->name('api.v1.products.index')
10 — CONFIG

Config Key Naming

DON'T
'otp_exp' // abbreviation 'jwt_ttl' // TTL = time to live (jargon) 'mail_drv' // abbreviation 'db_host' // acceptable but... 'CACHE_DRIVER' // SCREAMING_CASE (wrong convention)
DO
'otp_expiry_minutes' // descriptive, unit included 'jwt_access_lifetime' // full words, clear meaning 'mail_driver' // full word 'database_host' // full word 'cache_driver' // snake_case, lowercase
// Example config/auth.php — fully human-readable: return [ 'otp' => [ 'expiry_minutes' => 5, 'max_attempts' => 5, 'code_length' => 6, 'max_sends_per_hour' => 5, ], 'jwt' => [ 'secret' => env('JWT_SECRET'), 'access_token_lifetime' => 15, // minutes 'refresh_token_lifetime' => 43200, // minutes (30 days) ], 'allowed_channels' => ['email', 'sms', 'whatsapp'], 'default_channel' => 'email', ];
