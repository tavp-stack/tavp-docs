Overview

What is TAVP?

A curated tech stack that integrates four proven technologies into one cohesive developer experience, powered by Phalcon's C-extension engine.

🎨

Tailwind CSS

Utility-first CSS framework. Build any design directly in Volt templates without writing custom CSS. Responsive, composable, production-optimized.

⚡

Alpine.js

Lightweight JavaScript (15KB). Declarative interactivity via x-data, x-show, @click. Replaces jQuery. Eliminates the need for Livewire-like reactive layers.

📄

Volt

Phalcon's template engine, inspired by Jinja2 and Blade. Clean syntax, inheritance, macros, and automatic PHP compilation with built-in caching.

🔧

Phalcon

Full-stack MVC framework as a C-extension. Not PHP code — compiled C loaded into PHP engine. Zero file loading, zero autoloading per request.

Key Distinction

TAVP is a tech stack, not a framework. Phalcon is the framework (engine). TAVP is the integrated experience layer on top of Phalcon — like how TALL Stack combines Laravel with Tailwind, Alpine, and Livewire. But TAVP uses equal-weight components, with no single component dominating.

TAVP Coil

Swoole-powered runtime with native coroutines, async I/O, and persistent memory. Maximum performance for VPS and dedicated servers.

Traditional TAVP (PHP-FPM): Request → PHP-FPM Worker → Boot Phalcon → Process → Response → Kill (Every request: full framework boot) TAVP Coil (Swoole): ┌───────────────────────────────────────────────────────┐ │ Swoole Server Process (stay alive, never restarts) │ │ │ │ ┌─ Worker 1 ──────────────────────────────────────┐ │ │ │ Phalcon app LOADED ONCE, stays in memory │ │ │ │ ┌─ Coroutine 1: handle Request A ────────────┐ │ │ │ │ ├─ Coroutine 2: handle Request B ────────────┤ │ │ │ │ ├─ Coroutine 3: handle Request C ────────────┤ │ │ │ │ └─ Coroutine N: handle Request N ────────────┘ │ │ │ └──────────────────────────────────────────────────┘ │ │ │ │ ┌─ Worker 2 ──────────────────────────────────────┐ │ │ │ Phalcon app LOADED ONCE, stays in memory │ │ │ │ ┌─ Coroutine 1: handle Request X ────────────┐ │ │ │ │ ├─ Coroutine 2: handle Request Y ────────────┤ │ │ │ │ └─ ... ┘ │ │ │ └──────────────────────────────────────────────────┘ │ └───────────────────────────────────────────────────────┘

What Swoole Provides

  • ✓ Native coroutines — cooperative multitasking inside PHP processes
  • ✓ Async I/O — hooks blocking PHP functions at the C level, auto-converts to non-blocking
  • ✓ Persistent memory — Phalcon (C-extension) loaded once per worker, stays in RAM
  • ✓ 10,000+ concurrent coroutines per worker
  • ✓ Built-in connection pooling (MySQL, Redis, HTTP)
  • ✓ Built-in timer, task queue, WebSocket server

Coroutine Usage in Controllers

// Parallel API calls in one request public function index() { $users = null; $orders = null; $stats = null; Co\join([ go(function() use (&$users) { $users = $this->api->get('/users'); }), go(function() use (&$orders) { $orders = $this->api->get('/orders'); }), go(function() use (&$stats) { $stats = $this->api->get('/stats'); }), ]); // Total: 300ms (slowest call) // Sequential would be: 650ms return $this->view('dashboard', compact('users','orders','stats')); }

Benchmark: Swoole vs PHP-FPM

In comparable tests (Hyperf/Swoole vs Laravel/PHP-FPM), Swoole-based frameworks achieve ~15,000-19,000 requests/sec vs ~1,200-2,500 requests/sec for PHP-FPM — roughly 8-15x faster. Phalcon running inside Swoole has the potential to exceed this due to C-extension's lower per-request overhead.

Challenges & Solutions

Swoole long-lived processes introduce challenges that PHP-FPM never had. TAVP Coil addresses each one.

Memory Leaks

PHP-FPM auto-resets memory each request. Swoole does not. Variables that grow across requests will leak.

Solution: TAVP Coil RequestCleanup mechanism — resets request-scoped services, clears DI container entries, forces GC after each request.
State Pollution

Mutable state in service classes can "leak" from Request A to Request B. Superglobals ($_GET, $_POST) do not auto-reset.

Solution: TAVP enforces stateless service classes. All request data accessed via Phalcon Request object, never superglobals.
Database Connections

PHP-FPM auto-closes DB connections. Swoole keeps them open — can exhaust connection limits.

Solution: Swoole connection pool with configurable min/max connections. Connections returned to pool after each request.
Debugging Difficulty

Long-running processes make stack traces more complex. Errors in one coroutine can affect the whole worker.

Solution: Isolated error handling per coroutine. Structured JSON logging with request ID correlation. TAVP Coil debug mode with per-coroutine memory tracking.
File Handles

File handles stay open across requests. Can exhaust system limits under high load.

Solution: Explicit file close in finally blocks. TAVP's Storage abstraction auto-closes handles after use.
Not for Shared Hosting

Swoole requires CLI access and ability to run persistent processes. Not available on shared hosting.

Solution: TAVP Coil is for VPS/dedicated. Shared hosting uses standard PHP-FPM or TAVP Weave (Fibers).

TAVP Relay

RoadRunner-powered runtime with Go-based infrastructure. Go handles HTTP, queues, WebSocket, gRPC, and metrics. PHP handles business logic. Process isolation built-in.

TAVP Relay Architecture: ┌──────────────────────────────────────────────────────────┐ │ RoadRunner (Go binary) │ │ ├── HTTP server ............. Go handles all HTTP I/O │ │ ├── WebSocket server ........ Go handles connections │ │ ├── Queue worker ............ Go manages job pipeline │ │ ├── gRPC server ............. Go handles gRPC protocol │ │ ├── Metrics (Prometheus) .... Go exports metrics │ │ └── Scheduler ............... Go manages cron tasks │ │ │ │ │ │ communicates via unix socket / TCP │ │ ▼ │ │ ┌──────────────────────────────────────────────────────┐ │ │ │ PHP Workers (persistent, stay alive) │ │ │ │ ├── Worker 1: Phalcon app (in memory) │ │ │ │ ├── Worker 2: Phalcon app (in memory) │ │ │ │ ├── Worker 3: Phalcon app (in memory) │ │ │ │ └── Worker N: Phalcon app (in memory) │ │ │ └──────────────────────────────────────────────────────┘ │ │ │ │ Process isolation: PHP crash ≠ Go crash. │ │ If PHP worker dies, Go restarts it automatically. │ └──────────────────────────────────────────────────────────┘

What RoadRunner Provides

  • ✓ Process isolation — PHP crash does not take down the server
  • ✓ Go handles all I/O (HTTP, WebSocket, Queue, gRPC)
  • ✓ Automatic worker restart on failure
  • ✓ Built-in Prometheus metrics
  • ✓ Built-in job pipeline (no external queue needed)
  • ✓ GRPC for microservice communication
  • ✓ Single Go binary — no external dependencies

Why Not Just Swoole?

  • ! Swoole: one PHP error can crash entire worker (shared process)
  • ✓ Relay: Go isolates each PHP request (process boundary)
  • ! Swoole: all infra (queue, WS, metrics) must be configured separately
  • ✓ Relay: all infra built into single Go binary
  • ! Swoole: debugging long-running C-extension process is hard
  • ✓ Relay: Go handles infra, PHP only handles request logic

Inspired by Spiral Framework

Spiral Framework uses RoadRunner as its runtime and reports 5-10x faster performance than Laravel/Symfony. The key insight: "the ability not to unload the program core, database connections, and sockets from memory more than makes up for the limitations of persistent workers." TAVP Relay applies the same architecture to Phalcon.

CLI Usage

$ tavp relay:start TAVP Relay v1.0 (RoadRunner 2024.x) ├── Workers: 8 (PHP persistent) ├── HTTP: 0.0.0.0:8080 (Go) ├── Queue: enabled (Go job pipeline) ├── WebSocket: 0.0.0.0:8081 (Go) ├── gRPC: 0.0.0.0:9001 (Go) ├── Metrics: 0.0.0.0:2112 (Prometheus) └── Ready. $ tavp relay:start --workers=auto --relay=http,queue,websocket

TAVP Weave

PHP Fibers-based lightweight async. Zero external dependencies. Built into PHP 8.1+. Runs anywhere — including shared hosting. No Swoole, no RoadRunner required.

What PHP Fibers Provide

  • ✓ Cooperative multitasking — built into PHP 8.1+
  • ✓ Zero external dependencies (no extensions needed)
  • ✓ Works on shared hosting (PHP-FPM compatible)
  • ✓ Parallel I/O operations (HTTP calls, DB queries)
  • ✓ Synchronous-looking syntax (no callback hell)
  • ! Cannot hook blocking PHP functions (unlike Swoole)
  • ! Not true non-blocking I/O
  • ! Does not change PHP-FPM model (still process-per-request)

Usage in Controllers

use Tavp\Weave\Async; class DashboardController { public function index() { // Parallel fetch with Fibers $results = Async::all([ 'users' => function() { return User::query() ->where('active', true) ->get(); }, 'orders' => function() { return Order::query() ->where('status', 'pending') ->get(); }, 'stats' => function() { return $this->calculateStats(); }, ]); return $this->view( 'dashboard', $results ); } }

When to Use TAVP Weave

Perfect for shared hosting, small-to-medium apps, or any project where you want lightweight async without installing additional extensions. Use it for parallel HTTP calls to third-party APIs, parallel database queries, or any I/O-bound operations that benefit from concurrent execution. It won't turn PHP-FPM into an async server, but it will eliminate sequential wait times within a single request.

CLI Usage

# Enable TAVP Weave (zero config, just use it) $ tavp weave:enable TAVP Weave v1.0 ├── Runtime: PHP 8.3 Fibers (built-in) ├── Extension required: none ├── Works with: PHP-FPM, Apache mod_php, LiteSpeed └── Ready. Just use Async::all() in your controllers. # No separate server needed. Uses existing PHP-FPM. $ tavp serve # Regular PHP-FPM server with Fibers enabled

All Three Can Coexist

These are not mutually exclusive choices. A single TAVP project can use TAVP Relay as its production server (HTTP + Queue + WebSocket), TAVP Weave for parallel I/O inside controllers, and fall back to standard PHP-FPM on shared hosting. The application code stays the same — only the runtime changes.

Decision Guide
