1. Overview

Product Overview

📋

What is TAVP?

TAVP is a curated tech stack integrating Tailwind CSS + Alpine.js + Volt + Phalcon into one cohesive developer experience. Phalcon (C-extension) is the engine. TAVP is the car.

🎯

Product Purpose

This PRD defines every requirement needed to build TAVP from initial commit to a mature, self-sustaining ecosystem with community adoption and marketplace revenue.

📐

Versioning Strategy

ZeroVer: 0.MINOR.PATCH. Stay at 0.x during active development. Minor version = breaking changes. Patch version = backward-compatible fixes. 1.0.0 = stable, production-ready, no more breaking changes.

💡

Core Philosophy

Developer experience first. Performance as bonus. Modular architecture. Open source (MIT). No features locked behind paywall. Community-driven growth.

ZeroVer Reference

Per 0ver.org: "Software with major version zero is under initial development. Anything MAY change at any time. The public API SHOULD NOT be considered stable." TAVP follows this strictly — we stay at 0.x.y until the API is locked, tested, and documented for long-term stability.

Measurable Objectives

3. Target Audience

Who Will Use TAVP

👩‍💻

Primary: PHP Developers

Freelancers, small agencies, indie developers who want fast performance + good DX. Already familiar with Laravel/TALL or willing to learn. Value low hosting costs.

Need: Fast framework, good CLI, productive scaffolding, low hosting bill.

🏢

Secondary: Web Hosting Providers

Shared hosting companies, VPS providers, managed hosting. They benefit from Phalcon's low RAM footprint — more sites per server = more revenue.

Need: Pre-compiled Phalcon packages, easy installer, zero-cost integration.

🧑‍🎨

Tertiary: Non-Programmers

Small business owners, content creators, people who want a website but can't code. Use TAVP via visual builder + AI assistant + CMS.

Need: One-click install, visual editor, AI content generation, theme marketplace.

📱

Extended: Mobile Developers

Flutter/React Native developers who need a lightweight, fast backend API. TAVP's JWT + versioned API is their entry point.

Need: Clean REST API, JWT auth, OpenAPI docs, push notifications.

6. Concurrency

Three Concurrency Models

TAVP Coil (Swoole)

  • Native coroutines via Swoole C-extension
  • Async I/O: hooks all blocking PHP functions
  • Persistent memory: Phalcon booted once per worker
  • 10,000+ concurrent coroutines per worker
  • Connection pooling (MySQL, Redis, HTTP)
  • Built-in: timer, task queue, WebSocket server
  • Requires: Swoole extension + VPS/dedicated server
  • Risks: Memory leaks, state pollution, debug difficulty — all mitigated by RequestCleanup mechanism

TAVP Relay (RoadRunner)

  • Go-based application server handles HTTP, Queue, WebSocket, gRPC, Metrics
  • Process isolation: PHP crash does not take down Go server
  • Automatic worker restart on failure
  • Built-in job pipeline (no external queue needed)
  • Prometheus metrics export
  • Single Go binary — no external dependencies
  • Requires: RoadRunner binary + VPS/dedicated server
  • Inspired by: Spiral Framework (5-10x faster than Laravel/Symfony)

TAVP Weave (PHP Fibers)

  • Cooperative multitasking built into PHP 8.1+
  • Zero external dependencies (no extensions needed)
  • Works on shared hosting (PHP-FPM compatible)
  • Parallel I/O: HTTP calls, DB queries in one request
  • Synchronous-looking syntax (no callback hell)
  • Limitation: Cannot hook blocking functions (not true non-blocking)
  • Best for: Shared hosting, lightweight parallel operations

Coexistence

All three models share the same application code. A single project can use TAVP Relay as production server, TAVP Weave for parallel I/O inside controllers, and fall back to PHP-FPM on shared hosting. Only the runtime changes.

7. Versioning Strategy

ZeroVer: 0-Based Versioning

Per 0ver.org: "Software with major version zero is under initial development. Anything MAY change at any time."

ZeroVer Rules for TAVP: Format: 0.MINOR.PATCH MINOR (second number): Increment when: breaking changes, new major features, new ecosystem tools Examples: 0.1.0 → 0.2.0 (new features, possible breaking API changes) 0.2.0 → 0.3.0 (TAVPkit/TAVPhub launch, breaking module API) PATCH (third number): Increment when: bug fixes, documentation updates, backward-compatible changes Examples: 0.1.0 → 0.1.1 (bug fix in CLI installer) 0.1.1 → 0.1.2 (documentation update) When does TAVP reach 1.0.0? When ALL of these are true: ☐ Public API is locked (no more breaking changes without major version bump) ☐ Core framework passes 90%+ test coverage ☐ All ecosystem tools (TAVPkit, TAVPid, TAVPhub, TAVPblocks) are stable ☐ Documentation is complete (100+ pages) ☐ Module system API is stable (no breaking changes in module contracts) ☐ At least 10 community modules exist and work without modification ☐ At least 3 hosting providers support TAVP ☐ Performance benchmarks are published and reproducible Post-1.0 Rules: Follow strict SemVer (MAJOR.MINOR.PATCH) MAJOR = breaking changes (rare, planned well in advance) MINOR = new features (backward-compatible) PATCH = bug fixes

10. Risks & Mitigations

Known Risks

IDRiskImpactLikelihoodMitigation
R1 Phalcon C-extension hard to install on some environments High Medium Auto-installer with fallback chain. Pre-compiled packages at packages.tavp.dev. Universal installer script. cPanel/HestiaCP plugins.
R2 Phalcon project abandoned or development stalls High Low Phalcon has been active since 2012. TAVP provides abstraction layer that could theoretically swap engine. Monitor Phalcon release cadence.
R3 Small community, slow adoption Medium Medium Focus on DX first (Laravel-familiar syntax). Performance data as proof. WordPress migration guide. Hosting ecosystem for natural adoption.
R4 Swoole memory leaks in long-running processes Medium Medium RequestCleanup mechanism. Stateless service enforcement. Per-coroutine memory tracking in debug mode. TAVP Relay (RoadRunner) as alternative.
R5 Ecosystem fragmentation (too many competing community modules) Medium Low Official modules for core features. Module quality guidelines. Review system in marketplace. "Recommended" badges.
R6 PHP ecosystem shifts (major PHP changes break Phalcon) Low Low Phalcon tracks PHP releases. TAVP supports PHP 8.1-8.4. CI tests on all supported versions.
R7 AI integration costs are high for self-hosted users Low Medium Ollama driver for free self-hosted AI. TAVP Cloud free tier. AI features are optional, not core dependency.
